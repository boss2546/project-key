**Issue 2: `/api/reset`** ([main.py:2739](../../backend/main.py#L2739))
- Bulk delete all user data · iterate File rows · ลบ raw_path + db.delete
- **ไม่ลบ Drive · ไม่ remove vector_search**
- User กด "Reset" → sync ครั้งหน้า → **ทุกไฟล์งอกกลับ** เพราะ Drive ยังเก็บอยู่
- Critical bug — feature นี้แทบไม่ทำงานบน BYOS

**Issue 3: Sync Case A "stale link"** ([drive_sync.py:_pull_drive_to_local](../../backend/drive_sync.py))
- Flow ที่ crash:
  ```
  1. User connected · upload N files · cache rows มี drive_file_id ครบ
  2. User disconnect keep_files=False (main.py:3361)
     → cache rows ยังอยู่ · drive_file_id = NULL
  3. User reconnect → sync runs
  4. cache_files dict empty (keyed by drive_file_id NOT NULL)
  5. Drive listing returns N files with pattern {file_id}_*
  6. For each: not in cache_files → fall through
  7. v9.3.5.4 check: pattern + local_id IN all_local_file_ids → fall through to import
  8. _import_drive_file: db.add(File(id=local_id, ...)) → IntegrityError (duplicate id)
  ```
- v9.3.5.4 fix แค่จับ "orphan" (local_id NOT in cache) · ไม่จับ "stale link" (local_id IN cache · drive_file_id=NULL)

**Issue 4: Drive sub-folder orphans**
- ทุก file ที่ขึ้น Drive จะมี 1-3 entries:
  - `raw/{file_id}_<filename>` ← v9.3.5.3 พยายามลบแล้ว
  - `extracted/{file_id}.txt` ← **ไม่ได้ลบ** (ถ้ามี)
  - `summaries/{file_id}.md` ← **ไม่ได้ลบ** (ถ้ามี · เกิดหลัง organize)
- Sync iterate raw/ folder เท่านั้น · ไม่เคลียร์ sub-folder orphans
- Result: Drive folders บวมขึ้นเรื่อยๆ ตามจำนวน delete · user ไม่เห็นใน PDB UI แต่เปิด Drive จริงเห็น

```python
async def delete_extracted_text_from_drive_if_byos(
    user_id: str,
    db: AsyncSession,
    file_id: str,
) -> bool:
    """Best-effort: trash extracted/{file_id}.txt บน user's Drive.

    ใช้คู่กับ delete_drive_file_if_byos ตอน user delete file:
      raw/{file_id}_*  ← delete_drive_file_if_byos
      extracted/{file_id}.txt  ← THIS function
      summaries/{file_id}.md  ← delete_summary_from_drive_if_byos

    Why: เดิม delete ลบแค่ raw/ → extracted/ + summaries/ orphan สะสม
    """
    pair = await _get_byos_user_with_connection(user_id, db)
    if not pair:
        return False
    _user, conn = pair

    try:
        client = await _build_drive_client(conn)
        extracted_id = client.ensure_folder("extracted", parent_id=conn.drive_root_folder_id)
        filename = extracted_path_for(file_id).split("/")[-1]
        existing = client.find_file_by_name(filename, parent_id=extracted_id)
        if existing:
            client.delete_file(existing["id"])
            logger.info(
                "BYOS: trashed extracted/%s for user %s", filename, user_id
            )
            return True
        # No file = already gone · graceful no-op
        return False
    except Exception as e:
        # v9.3.5.5 — apply v9.3.5 graceful invalid_grant pattern
        if _is_refresh_failure(e):
            await _mark_drive_connection_errored(db, conn, e)
        logger.warning(
            "BYOS: delete extracted/%s failed for user %s: %s",
            file_id, user_id, e,
        )
        return False
```

### Step 4 — Apply same pattern ใน MCP `_tool_delete_file` (mcp_tools.py:1146)

แทนที่ existing function ด้วย:

```python
async def _tool_delete_file(db: AsyncSession, user_id: str, file_id: str) -> dict:
    """Delete a file and all related data — comprehensive cleanup (v9.3.5.5).

    Mirror DELETE /api/files/{id} 4-6 step cleanup so MCP delete behavior
    consistent with HTTP DELETE. ก่อน v9.3.5.5 path นี้ไม่ลบ Drive → file งอก.
    """
    if not file_id:
        raise ValueError("file_id is required")

    import os
    result = await db.execute(
        select(File).where(File.id == file_id, File.user_id == user_id)
    )
    file = result.scalar_one_or_none()
    if not file:
        return {"error": "File not found"}

    filename = file.filename
    drive_cleanup_attempted = False

    # 1. Disk
    if file.raw_path and os.path.exists(file.raw_path):
        try:
            os.remove(file.raw_path)
        except OSError as e:
            logger.warning(f"mcp delete_file: remove raw_path failed: {e}")

    # 2. Drive raw/ (v9.3.5.5)
    if file.drive_file_id:
        try:
            from .storage_router import delete_drive_file_if_byos
            await delete_drive_file_if_byos(user_id, db, file.drive_file_id)
            drive_cleanup_attempted = True
        except Exception as e:
            logger.warning(f"mcp delete_file: BYOS drive delete failed: {e}")

    # 2.5. Drive extracted/
    try:
        from .storage_router import delete_extracted_text_from_drive_if_byos
        await delete_extracted_text_from_drive_if_byos(user_id, db, file.id)
    except Exception as e:
        logger.warning(f"mcp delete_file: BYOS extracted delete failed: {e}")

    # 2.7. Drive summaries/
    try:
        from .storage_router import delete_summary_from_drive_if_byos
        await delete_summary_from_drive_if_byos(user_id, db, file.id)
    except Exception as e:
        logger.warning(f"mcp delete_file: BYOS summary delete failed: {e}")

    # 3. Vector search
    try:
        from . import vector_search as _vs
        _vs.remove_file(file.id, user_id=user_id)
    except Exception as e:
        logger.warning(f"mcp delete_file: vector remove failed: {e}")

    # 4. DB cascade
    await db.delete(file)
    await db.commit()

    return {
        "status": "deleted",
        "filename": filename,
        "file_id": file_id,
        "drive_cleanup": "attempted" if drive_cleanup_attempted else "skipped",
    }
```

```python
                # have failed · skip import + try cleanup
                try:
                    self._client.delete_file(drive_id)
                    logger.info(
                        "BYOS sync: cleaned up orphan Drive file %s (id=%s)",
                        drive_id, local_id,
                    )
                except Exception as e:
                    logger.warning(
                        "BYOS sync: orphan cleanup failed for %s (id=%s): %s",
                        drive_id, local_id, e,
                    )
                continue

            # A2 — STALE-LINK (v9.3.5.5): cache row exists but drive_file_id was cleared
            # (e.g., post-disconnect with keep_files=False) · UPDATE row instead of
            # creating duplicate (would IntegrityError)
            stale_row_q = await self.db.execute(
                select(File).where(
                    File.id == local_id,
                    File.user_id == self.user_id,
                )
            )
            stale_row = stale_row_q.scalar_one_or_none()
            if stale_row:
                stale_row.drive_file_id = drive_id
                stale_row.drive_modified_time = _parse_drive_time(drive_f["modifiedTime"])
                stale_row.storage_source = STORAGE_SOURCE_DRIVE_UPLOADED
                stats["relinked"] = stats.get("relinked", 0) + 1  # NEW stat
                logger.info(
                    "BYOS sync: re-linked stale row id=%s to drive_id=%s",
                    local_id, drive_id,
                )
            continue

        # A3 — GENUINE NEW (drive_picked or manual upload via Drive UI)
        self._import_drive_file(drive_f, stats)
    else:
        # CASE B — drift?
        # ... existing code unchanged ...
```

## ⚠️ Risks / Open Questions

### Risks (พร้อม mitigation)

**R1 — Performance: /api/reset ทำ 60+ Drive API calls serial**
- Impact: Medium · 60 files × ~2s = 2 นาที + อาจ timeout
- Mitigation: Frontend แสดง loading + progress (อาจต้องทำ background task in v9.4.x · ตอนนี้ accept)
- Worse case: timeout → response delayed · DB still committed (good)

**R2 — Drive API rate limits**
- Drive API: 1000 queries/100 seconds/user · 10000/day
- 60 deletes = 60 calls · OK
- Reset 60 files × 3 entries = 180 calls · still OK
- Bigger user (1000+ files) · /api/reset = 3000 calls → may hit limit
- Mitigation: log warning · accept partial fail · log "drive_files_trashed: <60"

**R3 — Sync stale-link race condition**
- Edge: 2 concurrent sync runs on same user
- Both see Drive file with pattern · both UPDATE same row
- Last-write-wins · no data corruption (drive_file_id same value · drive_modified_time minor diff)
- Mitigation: accept (rare)

**R4 — Drive sub-folder helpers add 2 calls per delete**
- Slowness: extra 2 × 1-2s = 2-4s per delete
- Mitigation: accept (delete is rare action) · log timing

**R5 — Backwards compat: existing files without summary on Drive**
- delete_summary_from_drive_if_byos returns False if not found · graceful · OK

### Open Questions

**Q1 — /api/reset response timing: 200 immediate vs wait?**
- **Default: WAIT** (current behavior · simpler) · stats accurate
- Alt: Return 202 Accepted + background task → frontend poll
- → Recommend: WAIT for v9.3.5.5 · revisit in v9.4.x with proper queue

**Q2 — Drive delete sub-folders: parallel or serial?**
- **Default: SERIAL** (extracted then summary) — Drive doesn't support batch trash easily
- Alt: asyncio.gather → 2x faster
- → Recommend: SERIAL · code simpler · save 1-2s isn't worth complexity

**Q3 — Should we also iterate sync extracted/summaries folders for orphan cleanup?**
- **Default: NO** · scope creep · text files small (KB) · quota not critical
- Alt: Add to sync logic · cleanup orphans there too
- → Recommend: NO for v9.3.5.5 · revisit if user reports Drive bloat

**Q4 — keep_files=False breaking change concern?**
- Pre-v9.3.5.5: would crash with IntegrityError on reconnect
- Post-v9.3.5.5: works correctly via stale-link path
- This is fix not breaking change ✅

**Q5 — vector_search.remove_file always called even when row didn't have index?**
- Yes · function is idempotent (no-op if not indexed) · OK to call always
